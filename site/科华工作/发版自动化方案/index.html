    🧩 自动构建发版统一管理方案¶

    1. 方案目标¶

    • 实现构建产物统一存放与版本管理；
    • 构建交由开发完成，运维只负责部署与回退操作；
    • 支持 OPS 平台一键部署/回退，具备可追溯性与一致性；
    • 解耦构建和部署，兼容 jar/class/sql 等多种包类型。

    目录结构统一如下：

    版本号 推荐格式：yyyyMMdd.序号，例如 20250716.01

    Text Only

    4. 权限控制与上传机制¶

    角色 权限 说明
    开发账号或CI账号 读写 /opt/releases/<系统> 可上传构建产物、创建新版本目录
    运维平台账号 可读写软链 执行部署、切换软链指向
    其他用户 只读 禁止写入或删除

    支持上传校验（推荐附带 md5 文件，自动比对）

    #!/bin/bash

APP=order-service
SYSTEM=erp
VERSION=$1

DEPLOY_DIR="/opt/releases/$SYSTEM/$APP"
TARGET_LINK="$DEPLOY_DIR/current"
NEW_VER="$DEPLOY_DIR/$VERSION"

if [ ! -d "$NEW_VER" ]; then
  echo "❌ 版本不存在: $NEW_VER"
  exit 1
fi

# 校验文件完整性（如有）
if [ -f "$NEW_VER/deploy.md5" ]; then
  echo "🔍 正在校验文件完整性..."
  cd "$NEW_VER" && md5sum -c deploy.md5 || exit 1
fi

echo "🚀 开始部署 $APP 版本: $VERSION"
ln -sfn "$NEW_VER" "$TARGET_LINK"

# 可选：执行服务重启
systemctl restart ${APP}.service

echo "✅ 部署完成。当前版本：$VERSION"

    6. 回退机制¶

    • 获取历史版本目录列表（根据创建时间或版本号排序）
    • 选择目标版本
    • 切换软链 current 指向历史版本目录
    • 重启服务

    可封装成统一命令 rollback.sh <app> <version> 或平台操作按钮。

    7. OPS 平台集成建议¶

    功能面板包括：

    • ✅ 当前版本查看
    • ⬆️ 上传产物入口（集成CI/页面）
    • 📌 选择目标版本部署
    • 🔁 一键回退上一个版本
    • 🔍 校验日志与校验状态
    • 🧹 保留最近 N 个版本，自动清理旧版本（配置化）

    8. 清理策略¶

    推荐每个服务目录下仅保留：

    • 当前软链指向版本
    • 最近 N（如 5）个历史版本
    • 可设置清理脚本定期执行：
    Bash
    1
2
3
    find /opt/releases/erp/order-service -maxdepth 1 -mindepth 1 -type d \
    ! -name "$(readlink /opt/releases/erp/order-service/current)" \
    -printf '%T@ %p\n' | sort -n | head -n -5 | cut -d' ' -f2- | xargs rm -rf

    ✅ 总结¶

    特性 是否具备 说明
    多应用支持 ✅ 多系统、多服务皆可适配
    多格式支持 ✅ 支持 jar/class/sql/yml 等混合
    自动部署 ✅ 可通过脚本或平台触发
    快速回退 ✅ 软链切换 + 服务重启
    目录规范统一 ✅ 避免混乱，便于管理
    安全可控 ✅ 权限、审计、校验机制完整
