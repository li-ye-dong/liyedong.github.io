    项目结构¶

    构建项目的方法有很多种，但最好的结构是一致、直接、没有意外的结构。

    许多示例项目和教程会根据文件类型（例如 CRUD、Router、Model）来划分项目，这对于微服务或范围较小的项目来说非常有效。然而，这种方法并不适合我们这个包含多个领域和模块的单体应用。

    我发现在这些情况下更具可扩展性和可进化性的结构受到 Netflix 的 Dispatch 的启发，并进行了一些细微的修改。

    异步路由¶

    首先，FastAPI 是一个异步框架。它被设计用于处理异步 I/O 操作，这就是它如此快速的原因。

    然而，FastAPI 并不限制你只能使用 <font style="color:rgb(31, 35, 40);">async</font> 路由，开发者也可以使用 <font style="color:rgb(31, 35, 40);">sync</font> 路由。这可能会让初学者误以为它们是相同的，但事实并非如此。

    I/O 密集型任务¶

    在底层，FastAPI 可以有效地处理异步和同步 I/O 操作。

    • FastAPI 在线程池中运行 <font style="color:rgb(31, 35, 40);">sync</font> 路由 阻塞 I/O 操作不会停止事件循环 执行任务。
    • 如果路由定义为 <font style="color:rgb(31, 35, 40);">async</font> ，则通过 <font style="color:rgb(31, 35, 40);">await</font> 定期调用它 并且 FastAPI 相信您只执行非阻塞 I/O 操作。

    需要注意的是，如果您违反该信任并在异步路由中执行阻塞操作，则事件循环将无法运行后续任务，直到阻塞操作完成。

    import asyncio
import time

from fastapi import APIRouter


router = APIRouter()


@router.get("/terrible-ping")
async def terrible_ping():
    time.sleep(10) # I/O blocking operation for 10 seconds, the whole process will be blocked

    return {"pong": True}

@router.get("/good-ping")
def good_ping():
    time.sleep(10) # I/O blocking operation for 10 seconds, but in a separate thread for the whole `good_ping` route

    return {"pong": True}

@router.get("/perfect-ping")
async def perfect_ping():
    await asyncio.sleep(10) # non-blocking I/O operation

    return {"pong": True}

    当我们调用时会发生什么：

    1. <font style="color:rgb(31, 35, 40);">GET /terrible-ping</font>
      1. FastAPI 服务器接收请求并开始处理
      2. 服务器的事件循环和队列中的所有任务将等待，直到 <font style="color:rgb(31, 35, 40);">time.sleep()</font> 完成
        1. 服务器认为 <font style="color:rgb(31, 35, 40);">time.sleep()</font> 不是 I/O 任务，因此它会等待直到它完成
        2. 服务器在等待期间不会接受任何新请求
      3. 服务器返回响应。
        1. 响应后，服务器开始接受新的请求
    2. <font style="color:rgb(31, 35, 40);">GET /good-ping</font>
      1. FastAPI 服务器接收请求并开始处理
      2. FastAPI 将整个路由 <font style="color:rgb(31, 35, 40);">good_ping</font> 发送到线程池，其中工作线程将运行该函数
      3. 在执行 <font style="color:rgb(31, 35, 40);">good_ping</font> 时，事件循环从队列中选择下一个任务并对其进行处理（例如接受新请求、调用数据库）
        • 独立于主线程（即我们的 FastAPI 应用程序），工作线程将等待 <font style="color:rgb(31, 35, 40);">time.sleep</font> 完成。
        • 同步操作仅阻塞侧线程，而不阻塞主线程。
      4. 当 <font style="color:rgb(31, 35, 40);">good_ping</font> 完成工作后，服务器向客户端返回响应
    3. <font style="color:rgb(31, 35, 40);">GET /perfect-ping</font>
      1. FastAPI 服务器接收请求并开始处理
      2. FastAPI 等待 <font style="color:rgb(31, 35, 40);">asyncio.sleep(10)</font>
      3. 事件循环从队列中选择下一个任务并对其进行处理（例如接受新请求、调用数据库）
      4. 当 <font style="color:rgb(31, 35, 40);">asyncio.sleep(10)</font> 完成后，服务器完成路由的执行并向客户端返回响应

    警告

    关于线程池的注意事项：

    • 线程比协程需要更多的资源，因此它们不像异步 I/O 操作那样便宜。
    • 线程池的线程数量有限，也就是说，你的线程可能会耗尽，导致应用运行缓慢。阅读更多（外部链接）

    CPU 密集型任务¶

    第二个警告是，非阻塞等待操作或发送到线程池的操作必须是 I/O 密集型任务（例如打开文件、数据库调用、外部 API 调用）。

    • 等待 CPU 密集型任务（例如繁重计算、数据处理、视频转码）是毫无价值的，因为 CPU 必须工作才能完成任务，而 I/O 操作是外部的，服务器在等待该操作完成时不执行任何操作，因此它可以转到下一个任务。
    • 由于 GIL 的存在，在其他线程中运行 CPU 密集型任务也是无效的。简而言之，GIL 一次只允许一个线程工作，这使得它对于 CPU 任务毫无用处。
    • 如果您想优化 CPU 密集型任务，您应该将它们发送给另一个进程中的工作人员。

    from datetime import datetime
from zoneinfo import ZoneInfo

from fastapi.encoders import jsonable_encoder
from pydantic import BaseModel, ConfigDict


def datetime_to_gmt_str(dt: datetime) -> str:
    if not dt.tzinfo:
        dt = dt.replace(tzinfo=ZoneInfo("UTC"))

    return dt.strftime("%Y-%m-%dT%H:%M:%S%z")


class CustomModel(BaseModel):
    model_config = ConfigDict(
        json_encoders={datetime: datetime_to_gmt_str},
        populate_by_name=True,
    )

    def serializable_dict(self, **kwargs):
        """Return a dict which contains only serializable fields."""
        default_dict = self.model_dump()

        return jsonable_encoder(default_dict)

    在上面的例子中，我们决定创建一个全局基础模型：

    • 将所有日期时间字段序列化为具有明确时区的标准格式
    • 提供一种方法来返回仅具有可序列化字段的字典

    解耦并重用依赖项。依赖项调用会被缓存¶

    但 <font style="color:rgb(31, 35, 40);">parse_jwt_data</font> 只被调用一次，即在第一次调用中。

    FastAPI 响应序列化¶

    您可能认为您可以返回与您的路线的 <font style="color:rgb(31, 35, 40);">response_model</font> 匹配的 Pydantic 对象来进行一些优化，但您错了。

    FastAPI 首先使用其 <font style="color:rgb(31, 35, 40);">jsonable_encoder</font> 将 pydantic 对象转换为 dict，然后使用您的 <font style="color:rgb(31, 35, 40);">response_model</font> 验证数据，然后才将您的对象序列化为 JSON。

    这意味着您的 Pydantic 模型对象被创建了两次：

    • 首先，当您明确创建它以从您的路线返回时。
    • 其次，FastAPI 隐式地根据 response_model 验证响应数据。

    设置数据库命名约定¶

    名称保持一致很重要。我们遵循以下规则：

    1. 小写蛇形
    2. 单数形式（例如 <font style="color:rgb(31, 35, 40);">post</font> ， <font style="color:rgb(31, 35, 40);">post_like</font> ， <font style="color:rgb(31, 35, 40);">user_playlist</font> ）
    3. 将相似的表用模块前缀分组，例如 <font style="color:rgb(31, 35, 40);">payment_account</font> ， <font style="color:rgb(31, 35, 40);">payment_bill</font> ， <font style="color:rgb(31, 35, 40);">post</font> ， <font style="color:rgb(31, 35, 40);">post_like</font>
    4. 跨表保持一致，但具体命名也可以，例如
      1. 在所有表中使用 <font style="color:rgb(31, 35, 40);">profile_id</font> ，但如果其中一些只需要创建者的个人资料，则使用 <font style="color:rgb(31, 35, 40);">creator_id</font>
      2. 所有抽象表都使用 <font style="color:rgb(31, 35, 40);">post_id</font> ，例如 <font style="color:rgb(31, 35, 40);">post_like</font> ， <font style="color:rgb(31, 35, 40);">post_view</font> ，但在相关模块中使用具体命名，例如 <font style="color:rgb(31, 35, 40);">chapters.course_id</font> 中的 <font style="color:rgb(31, 35, 40);">course_id</font>
    5. <font style="color:rgb(31, 35, 40);">_at</font> 后缀表示日期时间
    6. <font style="color:rgb(31, 35, 40);">_date</font> 日期后缀

from typing import Any

from pydantic import UUID4
from sqlalchemy import desc, func, select, text
from sqlalchemy.sql.functions import coalesce

from src.database import database, posts, profiles, post_review, products

async def get_posts(
    creator_id: UUID4, *, limit: int = 10, offset: int = 0
) -> list[dict[str, Any]]: 
    select_query = (
        select(
            (
                posts.c.id,
                posts.c.slug,
                posts.c.title,
                func.json_build_object(
                    text("'id', profiles.id"),
                    text("'first_name', profiles.first_name"),
                    text("'last_name', profiles.last_name"),
                    text("'username', profiles.username"),
                ).label("creator"),
            )
        )
        .select_from(posts.join(profiles, posts.c.owner_id == profiles.c.id))
        .where(posts.c.owner_id == creator_id)
        .limit(limit)
        .offset(offset)
        .group_by(
            posts.c.id,
            posts.c.type,
            posts.c.slug,
            posts.c.title,
            profiles.c.id,
            profiles.c.first_name,
            profiles.c.last_name,
            profiles.c.username,
            profiles.c.avatar,
        )
        .order_by(
            desc(coalesce(posts.c.updated_at, posts.c.published_at, posts.c.created_at))
        )
    )

    return await database.fetch_all(select_query)
